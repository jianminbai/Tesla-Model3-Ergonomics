# 第十一章 GitHub Pages 发布与项目运营

> 本章核心观点：这本指南不应停留在本地 Markdown 文件里。通过 GitHub Pages 发布，它可以成为一个长期维护、可搜索、可分享、可收集真实案例的开源知识库。

---

## 11.1 为什么要做成网站

单个 Word 或 PDF 有一个问题：内容一旦生成，就很容易停止更新。

但驾驶人体工程学是一个持续迭代的主题：

- 新案例会不断出现；
- 座椅调整实验会有新数据；
- 插图会不断优化；
- 决策树会越来越准确；
- 不同身高、体型、驾驶习惯的反馈会不断补充。

因此，最适合本项目的形态不是静态电子书，而是：

```text
GitHub 仓库
+
MkDocs 网站
+
Issue 案例收集
+
Release 版本管理
+
PDF/EPUB 后续导出
```

---

## 11.2 GitHub Pages 的作用

GitHub Pages 可以把仓库中的 Markdown 文档发布为网站。

好处：

- 可以在线阅读；
- 支持搜索；
- 支持手机浏览；
- 可以分享给其他 Model 3 车主；
- 每次 push 后自动更新；
- 文档和源文件保持一致。

项目发布后，访问地址通常是：

```text
https://jianminbai.github.io/Tesla-Model3-Ergonomics/
```

---

## 11.3 MkDocs 的作用

MkDocs 是一个静态文档网站生成工具。它适合技术文档、知识库和开源项目。

本项目使用 `mkdocs-material` 主题，原因是：

- 中文支持较好；
- 导航清晰；
- 搜索体验好；
- 支持代码块复制；
- 支持暗色模式；
- 适合长期文档维护。

本地预览：

```powershell
pip install mkdocs mkdocs-material
mkdocs serve
```

构建检查：

```powershell
mkdocs build --strict
```

## 11.5 内容版本管理

建议采用语义化文档版本。

```text
V0.1 项目骨架
V0.2 人体基础章节
V0.3 Model 3 专项分析
V0.4 症状决策树
V0.5 康复与办公椅联动
V0.6 实验系统与量化记录
V0.7 可视化插图体系
V0.8 GitHub Pages 发布优化
V1.0 首个完整可发布版本
```

每个版本建议都有：

- UPDATE_NOTES；
- 对应 commit；
- Roadmap 勾选；
- 必要时创建 GitHub Release。

## 11.6 Issue 如何收集真实案例

项目后续价值来自真实案例，而不是只来自作者个人经验。

建议通过 Issue 收集：

- 车型；
- 身高体重；
- 座椅设置；
- 不适位置；
- 驾驶时长；
- 调整动作；
- 调整后变化；
- 是否办公椅也类似；
- 是否出现麻刺、放射、无力等风险信号。

但要注意：

> Issue 里不能做医学诊断，只能做人体工程学分析和调整建议。

## 11.7 Release 节奏建议

建议节奏：

### V0.x

快速迭代内容结构，每次增加一块能力。

### V1.0

形成首个完整版本，包含：

- 10 章以上正文；
- 20 张以上 SVG 图；
- 完整症状决策树；
- 实验记录系统；
- GitHub Pages 在线网站。

### V1.1 以后

以真实案例和插图优化为主。

## 11.8 从个人案例到开源知识库

这个项目的起点是一个人的长期驾驶不适和座椅调整经验，但它不应停留在个人体验。

真正的开源价值在于：

```text
个人案例
↓
力学解释
↓
结构化记录
↓
通用决策树
↓
更多案例验证
↓
可持续知识库
```

这也是本项目和普通论坛帖最大的区别。

---

## 11.9 本章小结

GitHub Pages 发布不是形式工程，而是项目从“个人笔记”变成“公共知识库”的关键步骤。

V0.8 之后，本项目应具备：

- 可在线阅读；
- 可搜索；
- 可导航；
- 可提交 Issue；
- 可持续迭代；
- 可通过实验数据不断修正结论。

下一步应进入 V0.9：发布前质量检查，包括章节一致性、链接检查、术语统一、图文引用和 V1.0 发布计划。
